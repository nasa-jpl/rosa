# 깃허브 운영 규칙

## 목적

- 브랜치, 이슈, 커밋, PR 운영 방식을 하나의 문서로 통일한다.
- 이 문서는 작업 에이전트가 판단 기준으로 사용하는 규칙서다. 모든 항목은 조건과 규칙으로 명시한다.
- 작업 순서는 항상 `이슈 → 브랜치 → 커밋 → PR → 머지`로 고정한다.

## 적용 범위

- 본 저장소의 모든 기능 개발, 버그 수정, 문서/설정 변경 작업
- GitHub Issue 및 Pull Request 생성 전반

---

## 1. 브랜치 전략

### 1.1 브랜치 종류

| 브랜치 | 역할 | 직접 커밋 |
|---|---|---|
| `main` | 배포 기준 브랜치 | 금지 |
| `dev` | 통합 및 검증 브랜치. 모든 작업 PR의 머지 대상 | 금지 |
| `feat/<...>` | 기능 개발 | 허용 |
| `fix/<...>` | 버그 수정 | 허용 |
| `etc/<...>` | 문서/설정/운영성 변경 | 허용 |

### 1.2 브랜치 네이밍 규칙

**네이밍 제약**
- 소문자와 하이픈(`-`)만 사용한다.
- 공백, 언더스코어(`_`), 대문자는 사용하지 않는다.
  turtle_agent의 경우에 언더스코어는 하이픈으로 대체한다.

**뎁스 결정 규칙**

| 조건 | 형식 | 예시 |
|---|---|---|
| 특정 에이전트의 서브 기능 개발 | `feat/<agent_name>/<feature>` | `feat/turtle-agent/obstacle-store` |
| 에이전트와 무관한 단독 기능 개발 | `feat/<feature>` | `feat/cli-parser` |
| 버그 수정 | `fix/<issue-name>` | `fix/pose-logger-crash` |
| 문서/설정/운영 | `etc/<task-name>` | `etc/update-readme` |

### 1.3 머지 기준

- `feat/*`, `fix/*`, `etc/*` → `dev`로 머지한다.
- 릴리스 시점에만 `dev` → `main`으로 머지한다.
- `main`에 직접 반영된 긴급 수정은 `dev`에도 동일하게 반영한다.

## 2. 표준 작업 흐름

작업 순서를 반드시 아래 순서로 수행한다. 순서를 건너뛰지 않는다.

1. 이슈를 먼저 생성한다.
2. `dev` 최신 상태를 반영한다.
3. 목적에 맞는 새 브랜치를 생성한다.
4. 코드 수정 및 로컬 테스트를 수행한다.
5. `uv run ruff check src/`와 `uv run ruff format src/`를 실행해 lint를 통과시킨다.
6. 변경 파일만 스테이징 후 커밋한다.
7. 원격 브랜치로 푸시한다.
8. `dev` 대상 PR 초안을 작성하여 사용자 승인을 받는다.
9. 승인 후 `gh pr create`를 실행한다.
10. 리뷰 → 머지 → 브랜치 삭제.

### 2.1 사전 확인 명령어

```bash
git branch --show-current   # 현재 브랜치 확인
git status                  # 작업 트리 확인
git checkout dev
git pull origin dev         # 최신 dev 반영
```

## 3. 이슈 생성 규칙

### 3.1 핵심 원칙

- 이슈 생성은 본인 판단으로 수행한다.
- 모든 커밋/PR 이전에 해당 작업을 설명하는 이슈가 먼저 존재해야 한다.

### 3.2 이슈 생성 절차

1. 3.3 템플릿에 따라 이슈 본문을 작성한다.
2. `gh issue create`를 실행한다.
3. 생성된 URL을 팀 채널에 공유한다.

### 3.3 이슈 본문 템플릿

```md
## 배경
-

## 목표
-

## 대상 범위
-

## 작업 프로세스
1. dev 최신 반영
2. 브랜치 생성
3. 작업 및 로컬 테스트
4. ruff lint 통과
5. 커밋 및 푸시
6. PR 생성 및 머지

## 완료 기준
- [ ]
- [ ]
```

### 3.4 실행 예시

```bash
gh issue create \
  --title "feat: CLI 인자 파서 추가" \
  --label "feat" \
  --body "$(cat <<'EOF'
## 배경
- ...
EOF
)"
```

## 4. 커밋 규칙

### 4.1 커밋 전 체크

- `git status`, `git diff`로 변경 내용을 확인한다.
- 로컬 실행 테스트를 수행한다.
- `uv run ruff check src/`에서 오류가 없어야 한다.
- 변경 파일만 스테이징한다.

### 4.2 커밋 메시지

- 형식: `<type>: <변경 목적 요약>`
- `type`은 브랜치 prefix와 반드시 일치시킨다.

| type | 사용 시점 |
|---|---|
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `docs` | 문서 변경 |
| `etc` | 설정/운영 변경 |

```bash
git add <path>
git commit -m "feat: CLI 인자 파서 추가"
```

### 4.3 원격 푸시

```bash
git push -u origin HEAD
```

---

## 5. 로컬 품질 검사

CI를 사용하지 않는다. 커밋 전 아래 명령어를 로컬에서 직접 실행한다.

```bash
uv run ruff check src/    # lint 오류 확인
uv run ruff format src/   # 포맷 자동 수정
```

- `ruff check`에서 오류가 없는 상태로 커밋한다.
- 설정은 `pyproject.toml`의 `[tool.ruff]` 섹션을 따른다.

## 6. PR 생성 규칙

### 6.1 핵심 원칙

- **단일 PR 단일 주제**: 하나의 PR은 하나의 주제만 다룬다.
- PR 생성 전 제목/본문 초안을 사용자에게 공유하고 **명시적 승인** 후에 실행한다.
- 사용자 승인 이전에는 `gh pr create`를 실행하지 않는다.
- ruff lint를 통과한 상태에서만 PR을 생성한다.
- **자기완결 금지**: 이슈를 생성한 사람이 해당 PR을 직접 머지하지 않는다. 이슈 생성과 PR 종료(머지)는 반드시 서로 다른 팀원이 수행한다.

### 6.2 PR 본문 템플릿

```md
## 연관 이슈
- Refs #<번호>   (또는 Closes #<번호>)

## 변경 내용
- 핵심 변경 1
- 핵심 변경 2

## 테스트
- [ ] 로컬 실행 테스트 완료
- [ ] ruff lint 통과
- [ ] 회귀 영향 확인
```

- `## 변경 내용`은 불릿 중심으로 정리한다.
- `## 테스트`는 체크박스 목록으로 정리한다.
- 필요 시 `## 영향 범위` 섹션을 추가해 리스크를 명시한다.
- 최종 머지되는 PR에서만 `Closes #<번호>`를 사용한다. 중간 PR은 `Refs #<번호>`를 사용한다.

### 6.3 실행 예시

```bash
gh pr create --base dev --head HEAD \
  --title "feat: CLI 인자 파서 추가" \
  --body "## 연관 이슈
- Closes #12

## 변경 내용
- argparse 기반 CLI 파서 구현
- --help 출력 포맷 정리

## 테스트
- [x] 로컬 실행 테스트 완료
- [x] ruff lint 통과
- [x] 회귀 영향 확인"
```

## 7. 완료 체크리스트

### 7.1 이슈 생성

- [ ] 이슈 본문을 템플릿에 따라 작성했는가?
- [ ] 생성된 이슈 URL을 팀 채널에 공유했는가?

### 7.2 브랜치/커밋

- [ ] 브랜치 네이밍이 Section 1.2의 조건표를 따르는가?
- [ ] 최신 `dev`를 기반으로 작업했는가?
- [ ] 변경 파일만 스테이징하고 커밋했는가?
- [ ] 커밋 메시지의 type이 브랜치 prefix와 일치하는가?
- [ ] 로컬 실행 테스트를 수행했는가?
- [ ] `ruff check src/`가 오류 없이 통과했는가?
- [ ] 원격 브랜치에 푸시를 완료했는가?

### 7.3 PR 생성

- [ ] PR 제목/본문 초안을 사용자에게 확인받았는가?
- [ ] `dev` 대상 PR을 생성했는가?
- [ ] 단일 PR 단일 주제 원칙을 준수하는가?
- [ ] 연관 이슈를 본문에 명시했는가?
- [ ] ruff lint 통과 후 PR을 생성했는가?

---

## 8. 금지 사항

- 이슈 없이 커밋/PR을 진행하는 행위
- 사용자 승인 없이 `gh pr create`를 실행하는 행위
- 하나의 PR에 여러 주제를 섞는 행위
- `main`에 직접 커밋/푸시하는 행위
- 브랜치 이름에 대문자/공백/언더스코어를 사용하는 행위
- ruff lint 오류가 있는 상태로 PR을 생성하는 행위
- 이슈를 생성한 본인이 해당 PR을 직접 머지하는 행위 (자기완결 금지)
